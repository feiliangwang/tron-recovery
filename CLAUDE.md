## Build Commands

```bash
make build        # Build all components (scheduler, worker, bloomtool) → build/
make scheduler    # Build scheduler only
make worker       # Build CPU worker only
make worker-gpu   # Build GPU worker (requires CUDA, sm_75 arch)
make bloomtool    # Build Bloom filter tool
make test         # Run all tests: go test -v ./...
make clean        # Remove build/ directory
make deps         # go mod download && go mod tidy
```

Run a single test package:
```bash
go test -v ./internal/mnemonic/...
go test -v -run TestFunctionName ./internal/compute/...
```

## Architecture

Two-component distributed system communicating over HTTP:

**Scheduler** (`cmd/scheduler/`) — HTTP server (default :8080)
- Owns the mnemonic template with `?` placeholders for unknown words
- Manages job/task state, persisted to `cmd/scheduler/data/state.json`
- Distributes index ranges (not mnemonic lists) to workers
- Applies Bloom filter second-stage validation on submitted results
- Embeds a web UI for monitoring

**Worker** (`cmd/worker/`) — Computation client
- Fetches `[StartIdx, EndIdx)` index ranges from scheduler
- Locally converts indices → mnemonics via modulo arithmetic on BIP39 wordlist
- Runs full cryptographic pipeline per mnemonic
- Supports `-bench N` (benchmark mode) and `-verify N` (GPU vs CPU comparison)

### Cryptographic Pipeline (per mnemonic)

```
index → mnemonic (BIP39 wordlist modulo)
  → PBKDF2-HMAC-SHA512(2048 iter, empty passphrase)
  → BIP44 derivation: m/44'/195'/0'/0/0  (TRON path)
  → Keccak256(uncompressed pubkey, drop 0x04 prefix)
  → last 20 bytes = TRON address
```

### Communication Protocol

**v1**: JSON over HTTP (`internal/protocol/protocol.go`)
**v2**: Compact binary (`internal/protocol/compact.go`)
- `CompactTask`: 32 bytes `[TaskID:8][JobID:8][StartIdx:8][EndIdx:8]`
- `TaskTemplate`: sent once per job, contains unknown positions + known words
- Envelope: `[workerIDLen:1][workerID][payload]`

Workers send speed metrics (addresses/sec) with each task fetch; scheduler uses this for adaptive batch sizing.

## Key Internal Packages

| Package | Role |
|---|---|
| `internal/mnemonic` | BIP39 wordlist, validation, index-based enumerator |
| `internal/bip44` | HD wallet derivation (secp256k1, hardened keys) |
| `internal/crypto` | Keccak256 (SHA3 legacy variant) |
| `internal/compute` | `SeedComputer` interface; `cpu.go` and `gpu.go`/`gpu_fallback.go` implementations |
| `internal/bloom` | Bloom filter wrapper with gob serialization |
| `internal/scheduler` | `TaskManager`: job/task lifecycle, JSON persistence |
| `internal/protocol` | v1 JSON + v2 compact binary encoding/decoding |
| `internal/worker` | HTTP client (`http_client.go`, `compact_client.go`), worker loop |
| `internal/tron` | TRON HTTP API client for address verification |

## GPU Build Notes

GPU support uses CGo with a pre-compiled CUDA static library:
- Source: `internal/compute/gpu.cu`, bridged via `internal/compute/gpu_bridge.h`
- Built with `make worker-gpu`, requires CUDA toolkit at `/usr/local/cuda`
- CPU fallback: `internal/compute/gpu_fallback.go` (compiled without `cuda` build tag)
- Build tag: `-tags cuda` enables real GPU code

## State & Data Files

- `cmd/scheduler/data/state.json` — persisted job/task state (auto-recovered on restart)
- `*.gob` files — Bloom filter data (generated by bloomtool)
- Scheduler `-bloom` flag accepts either a `.gob` file or a raw address list `.txt` file
